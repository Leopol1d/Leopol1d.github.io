RequestMapping具体使用方法

@RequestMapping是Spring MVC框架中用来映射请求路径的注解，通过在Controller方法上使用@RequestMapping注解，可以指定该方法所映射的请求路径。具体使用方法如下：

  1. 用@RequestMapping注解标注Controller类或者Controller类中的方法。
  1. 在@RequestMapping注解中指定请求路径，可以使用相对路径或绝对路径，相对路径时基于类上指定的路径。
  1. 可以在@RequestMapping中指定请求方法，限制请求方式。
  1. 也可以使用简化的语法，使用@GetMapping、@PostMapping等注解，代替@RequestMapping注解。
  1. 在请求路径中使用变量，可以通过在路径中添加{变量名}的方式，然后在方法参数中添加@PathVariable注解来获取。
  1. 在请求中获取参数，可以使用@RequestParam注解来获取参数。如果请求参数名称和方法参数名称相同，则可以省略@RequestParam注解。

以上是@RequestMapping注解的一些常用用法，通过注解的方式可以方便地进行请求路径的映射和参数获取。

pom.xml文件中<scope>标签的作用

在 Maven 的 pom.xml 文件中，<scope> 标签用于指定依赖关系的作用范围。具体来说，<scope> 标签指定了依赖项在哪些阶段可用，以及它们是否应该包含在您的应用程序或库的构建中。

以下是 scope 属性的常用值及其含义：

  • compile：这是默认值，表示依赖项在所有阶段（编译、测试、运行）都可用，且应该包含在构建中。
  • provided：表示依赖项由运行时环境提供，例如 Servlet API 或 JPA API。这意味着在编译和测试期间需要依赖项，但在运行时不需要。这些依赖项不应该包含在构建中。
  • runtime：表示依赖项在运行时可用，但在编译时不需要。这通常是一些非必需的库，例如 JDBC 驱动程序。这些依赖项不应该包含在构建中。
  • test：表示依赖项仅在测试代码中可用，不应该包含在构建中。
  • system：表示依赖项是在本地文件系统上的某个地方手动安装的 JAR 文件。这通常是为了引用一些本地库，而这些库没有被 Maven 管理。这些依赖项不应该包含在构建中。
  • import：表示这个依赖项是由 <dependencyManagement> 元素中的一个 <dependency> 导入的，而不是直接在这个模块中定义的。

通过设置正确的 scope 属性，您可以确保只有必需的依赖项被包含在构建中，并且不会引入不必要的库。这有助于减少构建时间和应用程序的大小，提高应用程序的性能。

TTL (Time to Live)

TTL (Time to Live) 是一个网络术语，用于表示 IP 数据包在网络中传输的最大生存时间。具体来说，TTL 是 IP 数据包头部中的一个字段，其值表示数据包在网络中可以被路由器转发的最大次数，通常以跳数（hop count）的形式表示。每经过一个路由器，TTL 的值就会减 1。当 TTL 的值降到 0 时，该数据包就会被丢弃，以避免它在网络中永远循环。

TTL 的作用是防止数据包在网络中无限循环，因为如果一个数据包在网络中一直被路由器转发，但是没有到达其目的地，那么它将占用网络资源并影响其他数据包的传输。通过设置 TTL 值，数据包可以在网络中有限的跳数内到达目的地，从而确保网络资源的有效使用。通常，TTL 的值在网络中设置为 64 或 128。

Controller中RequestMapping中的路径

@RequestMapping("list")与@RequestMapping("/list")都能正常访问

@RequestMapping("list")和@RequestMapping("/list")在使用上没有本质区别，它们的作用是相同的，都是用于映射请求路径和处理方法之间的关系。

在Spring中，RequestMapping注解用于将HTTP请求映射到控制器中的处理方法。当使用@RequestMapping注解时，可以指定请求路径，以便当客户端发送该路径的请求时，可以找到对应的处理方法。在指定请求路径时，可以使用相对路径或绝对路径，相对路径指的是相对于当前处理方法所在的路径，绝对路径指的是从应用程序的根目录开始的完整路径。

在这种情况下，@RequestMapping("list")和@RequestMapping("/list")的唯一区别在于路径的开头是否带有斜杠"/"。当路径以斜杠开头时，它被解释为相对于应用程序的根目录的绝对路径。没有斜杠的路径被解释为相对于当前请求所在的路径的相对路径。因此，如果在处理方法中使用@RequestMapping("list")而不是@RequestMapping("/list")，则该路径将相对于处理方法的路径进行解释。这可能会导致路径解析错误，从而导致无法找到正确的处理方法。

因此，建议在@RequestMapping注解中始终使用绝对路径，以避免出现路径解析错误。这样，使用@RequestMapping("/list")会更可靠，因为它指定了一个相对于应用程序根目录的绝对路径。

@JsonIgnoreProperties({ "handler","hibernateLazyInitializer" })

"@JsonIgnoreProperties({ "handler","hibernateLazyInitializer" })" 是在使用 Spring Boot 和 Hibernate/JPA 进行开发时，用于解决返回 JSON 格式数据时，因为 Hibernate/JPA 的懒加载机制，而导致的问题。

具体来说，当我们从数据库中获取一个对象时，如果该对象有关联的子对象，那么在返回 JSON 格式数据时，会默认将该子对象也加载出来，而这会导致数据冗余和性能问题。而在使用 Hibernate/JPA 时，通常使用懒加载机制，即只有在访问子对象时才去加载。

所以，@JsonIgnoreProperties({ "handler","hibernateLazyInitializer" }) 注解的作用就是告诉 Spring Boot，在返回 JSON 格式数据时，忽略掉 handler 和 hibernateLazyInitializer 这两个属性，以解决数据冗余和性能问题。

Vue的生命周期

  1. beforeCreate（创建前）：在实例化 Vue 时调用，这时 data 和 methods 等都还未初始化，因此无法访问这些数据。
  2. created（创建后）：在实例化 Vue 后调用，此时 data 和 methods 等已经初始化完成，但是此时还没有开始编译模板。
  3. beforeMount（挂载前）：在模板编译完成后，即将挂载到页面前调用，此时 DOM 未渲染。
  4. mounted（挂载后）：实例已经挂载到页面，DOM 已经渲染完成，此时可以进行 DOM 操作。
  5. beforeUpdate（更新前）：响应式数据更新时调用，此时虚拟 DOM 将会重新渲染，但还未进行渲染。
  6. updated（更新后）：虚拟 DOM 重新渲染并应用到真实 DOM 后调用，此时页面已经被更新。
  7. beforeDestroy（销毁前）：在实例销毁之前调用，此时实例仍然可用。
  8. destroyed（销毁后）：在实例销毁后调用，此时所有绑定已解除，事件监听器已移除，子实例也已被销毁。

其中双冒号什么意思

双冒号是Java 8中的新特性之一，称为"方法引用"，用于简化Lambda表达式的语法。它可以理解为对某个方法的引用，代替了Lambda表达式中对某个方法的具体实现。

在Thymeleaf中，双冒号的使用和Java 8中类似，是用于引用某个片段的语法。例如，adminHeader::html指向一个名为"adminHeader"的片段的HTML文件，这个片段会被包含在当前页面的头部位置。

更具体地说，adminHeader::html表示：

  • "adminHeader"：片段的名称
  • ::：分隔符，表示片段名称和后面的模板文件名之间的分隔符
  • "html"：模板文件的扩展名

使用th:include指令可以在Thymeleaf模板中包含其他片段。这些片段可以在同一个模板中定义，也可以在其他模板文件中定义。通过使用双冒号来引用这些片段，可以避免在代码中直接写出片段的完整路径，从而使代码更加简洁和易于维护。

th:replaice和includeu有什么区别

在Thymeleaf模板引擎中，th:replace和th:include都可以用于将其他模板文件的内容嵌入到当前模板文件中。

但是，它们的实现方式有所不同：

  • th:replace指令会直接替换掉当前元素的整个内容，并将其他模板文件的内容替换到当前元素中，即只保留其他模板文件的内容，不保留当前元素的任何内容。
  • th:include指令则会将其他模板文件的内容嵌入到当前元素内部，即将其他模板文件的内容添加到当前元素的子节点中。

简而言之，th:replace是直接替换当前元素，而th:include则是将其他模板文件的内容添加到当前元素中。

在使用时，th:replace和th:include的语法也略有不同，具体可以参考Thymeleaf的官方文档。

在Java中，使用 static 关键字可以创建静态成员和静态内部类。静态内部类是嵌套在另一个类中且使用 static 修饰的内部类。

静态内部类与非静态内部类有一些区别和特点：

静态内部类可以直接访问外部类的静态成员和方法，而无需创建外部类的实例。 静态内部类不能直接访问外部类的实例变量和方法，除非通过创建外部类的实例进行访问。 静态内部类与外部类之间的关系不是紧密的，它们可以相互独立存在，即使外部类实例被销毁，静态内部类的实例仍然可以存在。 静态内部类可以有自己的静态成员变量、方法和内部类，它们与外部类的静态成员和内部类没有直接的关联。 静态内部类常见的使用场景包括：

将一个类封装在另一个类内部，以实现封装和隐藏。 创建辅助类或工具类，它们与外部类紧密相关但又不需要访问外部类的实例变量和方法。 优化代码结构，提高可读性和可维护性。 总结来说，静态内部类允许在一个类内部创建一个与外部类相互独立的类，具有自己的成员和方法，可以直接访问外部类的静态成员，但不能直接访问外部类的实例变量和方法。
